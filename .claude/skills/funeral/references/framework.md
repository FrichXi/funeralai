# 葬AI 分析框架 — 完整参考文档

## 核心理念

只关心一件事：**有没有真实的人在反复用这个东西，解决真实的问题。**

不是"市场前景如何"，不是"团队背景怎样"，不是"技术架构先进不先进"。

吹牛逼可以。但你得有一个过得去的产品。

## 铁律

**判断必须完全基于提交的材料内容，不基于对这家公司的已有认知。** 同一家公司在不同阶段可以有完全相反的评价。材料里说它好用就是好用，材料里说它不能用就是不能用。你不是在判断一家公司的永久价值，你是在判断这份材料里呈现的证据。

---

## 前置分类：广告检测

在做任何分析之前，先判断材料性质。

### 广告/软文特征

以下特征 = 广告/软文（advertorial）：

- 通篇夸，一个缺点都没提
- 满嘴"颠覆""引领""全球首个"，但没有一句话说产品具体怎么用
- 通稿结构：讲趋势 → 引出产品 → 展望未来
- "用户反馈"过于完美
- 作者明显没亲手用过，在转述官方话术
- 网站本身就是营销页：纯 landing page，没有产品入口、文档、定价页，全是热词和 CTA 按钮

判断为广告 → 仍然做分析，但标注哪些信息来自推广（可信度打折）。投资建议固定为"整不明白"。

### 不可评估判断

没有在评价具体产品（行业观察、方法论、趋势分析、整活、专访但没评价产品好坏）→ 不可评估（non_evaluable），给个简要说明即可。

注意例外：
- GitHub 仓库 + 技术实查报告 = 可评估。README 本身就是产品描述，技术实查报告提供了验证证据。
- 产品网站 + 体验实查报告 = 可评估。网站内容本身就是产品描述。

### 三档分类

| 分类 | 含义 |
|------|------|
| evaluable | 材料涉及可评价的具体产品 |
| non_evaluable | 不涉及产品评价 |
| advertorial | 广告/软文（仍然可评估，但推广信息可信度打折） |

## 四层分析框架

**核心原则：抓住一个点猛打，不要四平八稳勾清单。**

投资看长板，不看短板。一个产品长板足够长，安全隐患、体验粗糙、定价偏高都能忍——用户会用脚投票。反过来，六项全是60分的产品比有明显缺陷但一项95分的产品更危险，因为它什么都不是。

### 第零层：实查 — 东西到底存不存在？

第零层是前置验证层，在判断"有人在用吗"之前，先确认东西是不是真的。**如果第零层就暴露严重问题，直接倾向"吹牛逼呢"，不需要后面三层来救。**

第零层根据分析目标不同，检查重点不同：

#### GitHub 代码实查

针对 GitHub 仓库，检查代码是不是真代码：

- **代码占比**：代码占比低于 30% 要警惕。一个声称是"产品"的东西，如果大部分内容是 markdown 文档和 prompt 模板，那它不是产品，是文档集。
- **测试覆盖**：没测试的项目不配叫产品。有测试 = 作者至少在认真对待这个东西。
- **贡献者分布**：单人项目和"团队交付"是两回事。README 说"我们团队"但只有一个人在提交代码 = 叙事和现实不符。
- **代码采样**：是解决真实问题的业务逻辑，还是 prompt 模板打包？是有设计的工程，还是胶水脚本堆砌？
- **README 声称 vs 实际**：声称"600k LOC"但实际代码只有 12k 行，这不叫误差，叫吹牛。

技术实查报告中的红旗直接作为负面证据。代码占比、测试覆盖、贡献者分布是硬指标，不受文案影响。

#### Web 产品体验实查

针对产品网站，检查产品是不是真的在运行：

- **加载时间**：加载超过 5 秒、HTTP 错误、被 Cloudflare 拦截——连门都进不去的产品不配讨论。
- **产品入口**：有没有登录/注册入口、定价页、文档、API 文档、更新日志？如果全是 landing page 热词和 CTA 按钮，没有任何产品入口，那它不是产品，是 PPT。
- **声称与实际**：声称"all-in-one platform"但网站只有一个静态页面；声称"AI 驱动"但页面上没有任何 AI 交互入口——说和做不匹配。
- **产品痕迹**：文档、更新日志、定价页、API 文档、用户社区链接——这些是产品在运营的证据。只有 landing page 的东西大概率还在 PPT 阶段。
- **JS 错误与技术质量**：大量 JS 错误、资源加载失败、内部链接 404——连自己网站都维护不好的团队，产品质量可想而知。
- **交互元素**：声称是产品但网站上没有表单、输入框、按钮——这不是产品网站，是宣传册。
- **链接健康度**：内部链接全部 404 = 网站在烂。

体验实查报告中的响应时间、JS 错误数、交互元素数量、链接健康度是硬指标，不受文案影响。

#### 文章/文档材料

没有实查目标时，第零层不适用，直接从第一层开始。

### 第一层：唯一的硬指标 — 有人在用吗？

这是唯一能一锤定音的东西。其他都是辅助。

- **核心任务能不能稳定跑通？** 不是 demo 能跑，是日常用不崩。
- **有没有回头客？** 不是发布日围观一波就走，是有人在反复用。
- **用户动机是什么？** 是被产品本身吸引的，还是被补贴/热度/猎奇心态拉来的？

补充判断信号：
- **Stars 不等于用户。** 31k stars 可能只说明 README 写得好或者上了 Hacker News，不代表有人在用。看 issues 活跃度、PR 数量、fork 后的二次开发才是真使用信号。
- **网站流量信号要看仔细。** 有定价页、有文档、有更新日志 = 产品在运营。只有 landing page + "Join waitlist" = 还在画饼。

**有人在反复用 + 核心能跑通 = 这个产品就立住了。** 哪怕粗糙、哪怕贵、哪怕有安全问题。没有这一条，后面说什么都是空的。

### 第二层：长板有多长？

过了第一层之后，看这个产品有没有让人眼前一亮的地方：

- **场景打得透不透？** 是吃透了一个细分，还是什么都想做结果什么都不是？窄不丢人，窄而深更容易积累壁垒。
- **有没有自己的地盘？** 和大厂之间有没有边界？
- **有没有真的降低了什么门槛？** 让之前做不到的事情变得能做了？
- **交互和流程是否比技术名词更值钱？** 好产品不是功能多，是把一件事做得更省步骤、更少打断、更容易接回原工作流。
- **壳不是原罪。** 引擎可以通用，但壳层体验、分支管理、版本控制、导出和默认工作流，常常才是用户每天反复打开的东西。如果用户每天打开的是这个产品而不是底层引擎，就按产品价值来判。
- **入口和媒介形态本身就是产品价值。** 入口更顺手，往往比模型多一点能力更重要。
- **做得早、做得快、能定义版本，本身就是资产。** 谁先把一个版本做出来、做顺、并在一段时间里成为默认参考，谁就拥有版本定义权。
- **开源生态和可共创能力是真壁垒。** 便宜、能改、能接、有人一起做，往往比品牌包装更能积累长期壁垒。

**长板够长 = 整挺好，即使周围全是毛刺。**

### 第三层：吹的和做的差多远？

葬AI不反对吹牛逼。好产品该吹。但只有牛逼没有产品 = 炒作。

- **创始人光环、融资故事、媒体造势**，和产品实际体验之间差距有多大？
- 是在**放大一个真实存在的产品优势**，还是在**掩盖一个根本不能用的东西**？
- **热词密度和产品实质的比值**——热词越多、实质越少，越危险。
- **GitHub 特化**：README/文章中的声称 vs 仓库实际数据，差距越大越危险。声称"all-in-one platform"但代码只有 3 个文件、声称"production-ready"但零测试、声称"团队"但一个人在写——这些是硬伤。
- **Web 特化**：页面声称 vs 网站实际状态，差距越大越危险。声称"millions of users"但网站只有一个静态页、声称"enterprise-grade"但 JS 报错满天飞、声称"AI-powered"但页面上没有任何 AI 交互——这些是硬伤。

## 三档结论标准

| 结论 | 条件 |
|------|------|
| **整挺好** | 第零层没有致命红旗，第一层过了（产品能用 + 有回头客），长板明确。短板多不怕，长板够长就行。 |
| **吹牛逼呢** | 第零层就暴露了（代码不是真代码 / 网站打不开 / 纯 landing page / 声称与现实严重不符），或者第一层没过（产品跑不通、没人在用、伪需求、纯炒作）。 |
| **整不明白** | 这是**默认值**。正负混杂看不清、信息太少、广告/软文、网站被反爬拦截无法获取内容——都判整不明白。宁可保守。 |

信息完整度分三级：high / medium / low。

## 证据分类与要求

### 证据类型

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| fact | 可验证的事实：用户数据、功能描述、定价、时间线、提交者亲测反馈 | 通用 |
| inference | 基于事实的推断 | 通用 |
| risk | 风险信号：信息缺失、逻辑矛盾、执行隐患 | 通用 |
| promotional | 推广/营销信息（可信度打折） | 通用，advertorial 材料必用 |
| code_inspection | 代码实查发现的硬证据 | 仅 GitHub 分析 |
| product_testing | 产品体验实查发现的硬证据 | 仅 Web 分析 |

### 证据要求

- **quote 必须是原文**，不要编造
- 每条证据的 claim 用一句话说清楚这条证据说明了什么
- **提交者一手体验是最高优先级证据**，标记为 fact
- 总共 5-8 条证据，抓最关键的，不要面面俱到
- 当页面声称和实际检查结果矛盾时，以检查结果为准
- 代码实查 / 产品体验实查报告中的数据是硬证据，优先级高于文案描述

### 证据优先级（从高到低）

1. 提交者一手体验（亲测反馈）
2. 实查硬数据（代码占比、测试覆盖、加载时间、JS 错误数等）
3. 可验证事实（用户数据、功能描述、定价等）
4. 基于事实的推断
5. 推广/营销信息（可信度打折）

## 产品概述要求

用 2-3 句大白话说清楚这个产品到底是什么，以及到底行不行。

- 抛开话术、愿景、热词，用最朴素的语言
- **不是中性描述——要把判断揉进去**
- 如果产品有明显的好或不好，直接说
- 如果有提交者一手体验，必须体现在描述中
- 如果有代码实查或产品体验实查数据，用一句话总结实查结论

### 好坏对比

差的产品概述（太中性，只描述不判断）：
> 一个AI生成可交互H5内容的UGC社区，通过单列Feed流分发，支持图像、语音、视频、3D等全模态。

好的产品概述（带态度，揉进了判断）：
> 一个AI生成互动H5的UGC社区，创始人定位"可以玩的抖音"——但实际体验更像AI版4399，完成度远未达到定义品类的水平。

## 常见误判模式

遇到以下模式要警惕，不要被表面功夫带跑：

| 模式 | 特征 | 倾向判断 |
|------|------|----------|
| **概念大于产品** | 主要在讲愿景和趋势，很少出现真实使用细节，关键任务如何完成说不清楚 | 整不明白 |
| **换名词当创新** | 旧东西换新名词，只改定语不改能力，创新主要体现在宣传词 | 吹牛逼呢 |
| **表演式复杂** | 界面做得很忙像在干活，用户每步都要反复确认，结果卡在长报告或残缺工作流里 | 吹牛逼呢 |
| **讲人不讲产品** | 创始人故事远多于产品体验，外部背书强于内部证据，产品价值要靠神性补 | 整不明白 |
| **供给幻想** | 默认相信复杂供给会自然涌现，把供给质量问题误写成民主化创作 | 整不明白 |
| **主体性神话** | 把交互效果直接神化成真正主体性，用具身、自主、生命感替代能力说明 | 吹牛逼呢 |
| **故事掩盖执行缺口** | 大词很多但过程控制、版本控制、可用性都没做好，从头到尾没有稳定可验证的任务闭环 | 吹牛逼呢 |
| **Stars 幻觉** | 高 stars 低实质。README 精美、Demo 能跑，但代码仓库里没有真正的产品代码。Stars 买得到、蹭热度蹭得到 | 看代码占比和测试覆盖 |
| **Landing Page 幻觉** | 网站做得漂亮、热词满满、有名人背书，但点进去没有产品入口、没有文档、没有定价。网站本身是个广告，不是产品 | 看有没有实际产品痕迹 |

## 输出纪律

- 先给结论，再展开证据，不要先写一篇中规中矩的综述
- 不要平均分配信号，单一强场景的证据可以压过一堆泛泛批评
- 必须区分事实、推断和风险，不允许发明证据
- 如果材料不够支撑明确判断，宁可输出整不明白，也不要写体面废话
- 实查报告中的数据是硬证据，优先级高于文案描述；当声称和实际矛盾时，以实际为准
- 读懂作者的整体态度——葬AI风格犀利，善用讽刺，经常一边骂一边夸。判断方向看总体态度，不是数负面段落。标题往往就是作者的核心判断
- 一篇文章提到多个产品时，聚焦标题指向的那个

## 参考案例

以下案例展示的是**证据模式 → 判断**的推理方式，不是对特定公司的固定评价。用于校准判断尺度。

### 案例 1：整挺好（文章评测）

**证据模式**：作者评测一个开源聊天客户端。材料显示：(1) 作者本人每天在用，粘性已形成；(2) 核心功能（分支和版本管理）稳定好用；(3) 打透了一个具体场景（写作工作流）；(4) 标题用"伟大"，整体态度明确正面。

**判断**：整挺好
**推理**：有真实重复使用证据 + 核心任务稳定跑通（第一层过了）；场景虽窄但打得透，壳层体验本身就是产品价值（第二层长板明确）。

### 案例 2：吹牛逼呢（文章评测）

**证据模式**：作者实测一个自称"全球首个 OS"的产品。材料显示：(1) 核心任务跑了50分钟完成度不到30%；(2) 第二个任务根本没进入目标页面，生成了错误的空文件；(3) 任务完成后没法交互；(4) 宣传语和实际产品形态严重不符（说OS实际是浏览器）。

**判断**：吹牛逼呢
**推理**：核心任务明确跑不通（第一层没过）；宣传与产品严重脱节，属于"换名词当创新 + 故事掩盖执行缺口"的误判模式。

### 案例 3：吹牛逼呢（GitHub 代码实查）

**证据模式**：一个 GitHub 仓库，README 声称是"all-in-one developer platform"，31k stars。技术实查报告显示：(1) 代码占比 25%，大部分是 markdown prompt 模板；(2) 零测试文件；(3) 单一贡献者主导 89% 提交；(4) 代码采样显示主要是 prompt 模板打包成 CLI 命令。

**判断**：吹牛逼呢
**推理**：第零层就暴露了——声称是 platform 但代码仓库里没有平台代码，是 prompt 模板集打包。Stars 数量和产品实质严重不匹配。代码占比、测试覆盖、贡献者分布全是红旗。

### 案例 4：整挺好（Web 产品体验实查）

**证据模式**：一个 SaaS 产品网站。体验实查报告显示：(1) 网站加载 800ms，HTTP 200；(2) 有完整的登录/注册入口、定价页、API 文档、更新日志；(3) 42 个表单/输入元素，说明有真实产品界面；(4) 内部链接全部健康。页面内容显示明确的使用场景和定价。

**判断**：整挺好
**推理**：第零层过了——网站健康、有完整产品基础设施；页面内容有明确场景和定价，不是画饼。

### 案例 5：吹牛逼呢（Web 产品体验实查）

**证据模式**：一个声称是"AI platform"的产品网站。体验实查报告显示：(1) 网站只有一个 landing page，大量热词但无产品入口；(2) 零表单/输入框，只有"Join waitlist"按钮；(3) 5 个 JS 控制台错误；(4) 无文档、无定价、无更新日志。

**判断**：吹牛逼呢
**推理**：第零层就暴露了——声称是 platform 但网站上没有产品入口，是纯营销页。无交互元素、无产品运营痕迹。

### 案例 6：整不明白（不可评估）

**证据模式**：作者讨论某个领域的战略路线之争。材料显示：(1) 提到多个产品但没有聚焦评价任何一个；(2) 核心论点是行业趋势判断而非产品好坏；(3) 没有亲手使用任何产品的体验描述。

**判断**：整不明白（non_evaluable）
**推理**：材料是行业观察不是产品评测，没有针对单一产品的使用证据，不构成投资判断依据。

### 案例 7：整不明白（反爬拦截）

**证据模式**：一个产品网站被 Cloudflare 拦截，HTTP 403。无法获取页面内容，无法进行体验测试。

**判断**：整不明白
